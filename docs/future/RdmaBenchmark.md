# RDMA Benchmark Server & Client

gRPC benchmark binary pair for cross-node performance testing over RDMA.

**Status:** Phase 1 implemented (TLS mode). See `tests/rdma-io-bench/`.

## Overview

Two binaries in the `rdma-io-bench` crate (`tests/rdma-io-bench/`):

- **`rdma-bench-server`** — gRPC `SayHello` service over RDMA+TLS
- **`rdma-bench-client`** — configurable threads, connections, duration;
  collects latency percentiles (p50/p99/p999) and throughput

## Protocol Modes

| Mode | Flag | Transport Stack | RDMA Transport |
|------|------|-----------------|----------------|
| `tls` | `--mode tls` (default) | RDMA → TLS (OpenSSL) → HTTP/2 → gRPC | `TransportConfig::stream()` |
| `h3` | `--mode h3` (future) | RDMA → QUIC (Quinn) → HTTP/3 → gRPC | `TransportConfig::datagram()` |

## Usage

```bash
# Build
cargo build -p rdma-io-bench --release

# Server
rdma-bench-server --bind 0.0.0.0:50051 --cert certs/cert.pem --key certs/key.pem

# Client
rdma-bench-client --connect 10.0.0.1:50051 \
    --connections 4 --threads 2 --duration 10 --payload 64 \
    --cert certs/cert.pem --report json
```

## Output

Text format shows throughput + latency percentiles. JSON format for
automated collection:

```json
{
  "mode": "tls", "transport": "stream",
  "connections": 4, "threads": 2,
  "total_requests": 1254320, "throughput_rps": 125432.0,
  "latency_us": { "p50": 28.3, "p99": 95.1, "p999": 412.7 },
  "errors": 0
}
```

## TLS Certificates

Self-signed cert (no CA). Generated by `deploy_bench.yml` Ansible playbook
in `build/certs/` (gitignored). Client uses direct-trust: adds server cert
to trust store. Cert includes IP SANs for VM addresses.

## Ansible Integration

| Playbook | Purpose |
|----------|---------|
| `deploy_bench.yml` | Generate certs (if missing), copy binaries + certs to VMs |
| `bench_run.yml` | Parameterized: server on VM1, client on VM2, saves JSON results |

Entry point: `tests/e2e/run_bench.sh`

```bash
# Setup VMs first
./tests/e2e/run_tests.sh playbooks/setup_rdma.yml
./tests/e2e/run_tests.sh playbooks/deploy_bench.yml

# Single run
./tests/e2e/run_bench.sh --mode tls --connections 4 --threads 2

# Full matrix
./tests/e2e/run_bench.sh --matrix
```

## Key Implementation Details

- **Payload via name padding:** `HelloRequest.name = "x".repeat(payload_size)`
  (reuses existing proto without modification)
- **Per-connection histogram:** Each connection has its own `hdrhistogram`,
  merged at report time (no lock contention during benchmark)
- **Direct-trust TLS:** Client adds server cert to OpenSSL trust store,
  ALPN `h2` set on both sides
- **Workspace deps:** Uses `foo.workspace = true` for shared dependencies
- **H3 client needs `connect_to()`:** RDMA pre-connection step before QUIC
  handshake (see `tonic_h3_tests.rs` for pattern)
- **Tracing:** Uses `RUST_LOG` env var. Per-request errors are at `trace`
  level (won't flood), task failures at `warn`. Examples:
  ```bash
  # Local: set RUST_LOG directly
  RUST_LOG=info rdma-bench-client --connect ...
  RUST_LOG=rdma_io_bench=trace rdma-bench-client --connect ...

  # Ansible: pass bench_rust_log variable
  ./tests/e2e/run_bench.sh --mode tls -e "bench_rust_log=debug"
  ```

## VM Resources for Benchmarking

Default test VMs (2GB RAM, 2 vCPU) work but benchmarking benefits from more:

```bash
TF_VAR_vm_memory_mb=4096 TF_VAR_vm_vcpus=4 ./tests/infra-local/scripts/start-vm.sh -y
```

KVM required for meaningful results — software emulation is too slow.
